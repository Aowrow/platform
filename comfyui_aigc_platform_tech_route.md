# 基于 ComfyUI 的内部 AIGC 平台技术路线建议

## 1. 项目定位

这个平台的前提很明确：

- 使用对象是 **10 人左右的小团队**
- 平台性质是 **内部使用平台**，不是对外商业化产品
- `ComfyUI` 主要作为 **远端推理服务** 使用，平台通过 API 调用它
- 工作流的导出、编辑、维护主要放在 `ComfyUI` 内完成，**平台侧不做工作流编辑器**

因此技术路线应该遵循一个原则：

**能简单就简单，能少一层就少一层，优先保证稳定、易维护、易上手。**

不建议一开始就引入面向大型公网平台的复杂设计，比如：

- 积分/套餐/订单系统
- 模型管理中心
- 多租户体系
- 复杂的工作流管理后台
- 过重的微服务拆分

这个项目更适合采用：

**前后端分离 + 单体业务后端 + ComfyUI API 调用 + MySQL/Redis/对象存储**

---

## 2. 推荐总体架构

建议采用 4 层结构：

### 2.1 前端层

负责：

- 任务创建
- 参数填写
- 文件上传
- 任务列表
- 结果查看
- 简单的系统管理页面

推荐：

- `Next.js`
- `TypeScript`
- `Tailwind CSS`
- `shadcn/ui`
- `TanStack Query`
- `Zustand`

这样做的优点：

- 前端开发效率高
- 适合做内部工具型界面
- 页面组织清晰，后续扩展图片、视频、音频任务页都比较方便

### 2.2 业务后端层

负责：

- 用户登录与基础权限
- 任务创建与状态维护
- 调用 ComfyUI API
- 接收执行结果并落库
- 文件元数据管理
- 对前端提供统一 API

推荐：

- `NestJS`
- `TypeScript`
- `Prisma`
- `MySQL`
- `Redis`

为什么推荐 `NestJS`：

- 对小团队来说足够规范，也不会太重
- 模块化清晰，适合内部平台长期维护
- 前后端统一 `TypeScript`，沟通和开发成本低

### 2.3 推理接入层

这一层不需要做成特别复杂的“AI 中台”，更适合做成后端中的一个模块或一个轻量服务。

主要职责：

- 根据任务类型选择要调用的 ComfyUI 工作流接口
- 组装请求参数
- 调用 ComfyUI API 提交任务
- 查询任务执行状态
- 拉取或接收结果
- 把状态同步到平台数据库

这里的关键点是：

**平台不负责工作流编辑，只负责使用已经在 ComfyUI 中准备好的工作流。**

也就是说，平台更像：

- 参数填写层
- 任务调度层
- 结果管理层

而不是工作流设计器。

### 2.4 数据与文件存储层

推荐：

- `MySQL 8.x`
- `Redis`
- `MinIO` 或云对象存储

职责划分：

- `MySQL`：存业务数据、任务数据、素材索引、结果索引
- `Redis`：存缓存、临时状态、简单队列
- `MinIO`：存原始素材、生成结果、缩略图、预览文件

注意：

**图片、视频、音频文件不要直接存到 MySQL。**

## 3. 最推荐的技术栈组合

如果以“小团队内部平台、快速落地、后续可维护”为目标，我最推荐：

### 前端

- `Next.js`
- `TypeScript`
- `Tailwind CSS`
- `shadcn/ui`
- `TanStack Query`
- `Zustand`

### 后端

- `NestJS`
- `Prisma`
- `MySQL`
- `Redis`

### 推理端

- `ComfyUI`
- 平台通过 `HTTP API / WebSocket` 调用 ComfyUI

### 文件与部署

- `MinIO`
- `Nginx`
- `Docker Compose`

这是一个对小团队非常合适的组合：

- 技术栈统一
- 上手快
- 运维压力小
- 足够支撑内部图片/视频/音频生成平台

## 4. 为什么不需要做得太重

结合你的场景，下面这些能力可以先删掉：

### 4.1 不需要积分、套餐、订单系统

因为是内部平台，不对外收费，所以没必要建设：

- 积分账户
- 消费流水
- 订单系统
- 套餐权限
- 支付相关能力

这些都会明显增加复杂度，但对当前场景价值很低。

### 4.2 不需要模型管理中心

因为你这里的模型主要由 `ComfyUI` 所在的推理端管理，平台本身并不负责本地模型加载，也不直接在平台服务器上跑模型。

所以平台侧通常不必专门建：

- 模型表
- LoRA 表
- VAE 表
- 模型版本管理后台

除非未来你确实需要：

- 统计某个工作流对应哪个模型
- 限制某些用户只能使用某些推理能力
- 记录远端 ComfyUI 的模型配置快照

否则完全可以先不做。

### 4.3 不需要工作流管理后台

既然你已经决定：

**工作流的导出和维护直接在 ComfyUI 内完成**，

那么平台侧就不需要再建设：

- 工作流编辑器
- 工作流导出页面
- 节点级可视化配置页
- 工作流模板发布后台

平台只需要保存少量映射关系，例如：

- 某个“业务能力”对应哪个 ComfyUI 工作流
- 该工作流需要哪些前端参数
- 参数如何传给 ComfyUI

### 4.4 不需要一开始就拆微服务

你的团队规模和平台规模都不大，建议：

- 前端一个项目
- 后端一个项目
- ComfyUI 独立部署

不要一开始就拆成：

- 用户服务
- 任务服务
- 文件服务
- AI 编排服务
- 审计服务

这会增加联调、部署、排错和维护成本。

```text

      |
      v

      |

      v                        v
```

这个架构已经足够支撑内部平台。

如果后续任务量变多，再在后端中补一个异步任务队列模块即可，不需要现在就过度设计。

## 6. 核心模块建议

结合你的场景，我建议平台只保留下面这些核心模块。

### 6.1 登录与基础权限

内部平台不需要太复杂，保留最基础的即可：

- 登录/退出
- 用户管理
- 角色区分（普通用户 / 管理员）
- 接口鉴权

如果团队统一使用企业内部账号体系，后续还可以接入：

- LDAP
- OAuth2
- 企业微信 / 钉钉 SSO

但第一版不一定要上。

### 6.2 任务中心

这是平台最重要的模块。

建议负责：

- 创建任务
- 查看任务状态
- 展示执行进度
- 失败重试
- 查看历史记录
- 查看结果文件

任务类型建议统一抽象为：

- `image`
- `video`
- `audio`

状态建议统一为：

- `pending`
- `queued`
- `running`
- `success`
- `failed`
- `cancelled`

这样前端展示和后端处理都会更清晰。

### 6.3 ComfyUI 调用适配模块

这个模块负责把平台参数转换成 ComfyUI 能识别的请求。

建议职责：

- 维护不同业务功能对应的 ComfyUI 接口配置
- 组织提交参数
- 处理任务提交、轮询、回调
- 统一错误处理
- 统一状态转换

注意：

这里做的是 **接口适配**，不是工作流编辑。

### 6.4 文件与结果管理模块

建议负责：

- 上传输入素材
- 保存生成结果
- 记录文件元数据
- 提供预览地址
- 按任务关联素材和结果

建议把：

- 原图
- 生成图
- 视频文件
- 音频文件
- 缩略图

都统一收口到对象存储中。

### 6.5 系统配置模块

因为是内部平台，建议增加一个轻量配置模块，用来维护：

- ComfyUI 服务地址
- 超时时间
- 默认轮询间隔
- 文件存储配置
- 支持的业务类型开关

这样以后环境迁移、地址切换会方便很多。

## 7. MySQL 数据库设计建议

既然已经明确不需要计费、模型中心、工作流后台，那么数据库可以收敛到非常实用的一组表。

建议保留这些核心表：

- `users`：用户表
- `roles`：角色表
- `user_roles`：用户角色关联表
- `tasks`：任务主表
- `task_logs`：任务状态日志表
- `assets`：文件资源表
- `system_configs`：系统配置表

如果后续确实有必要，再补：

- `api_keys`：内部接口密钥表
- `audit_logs`：审计日志表

### 7.1 `tasks` 表建议字段

建议至少包含：

- `id`
- `task_type`：`image/video/audio`
- `biz_type`：具体业务类型，例如文生图、图生视频、TTS
- `user_id`
- `status`
- `input_params`：JSON
- `comfy_task_id`：ComfyUI 返回的任务 ID
- `result_asset_id`：结果资源 ID
- `error_message`
- `started_at`
- `finished_at`
- `created_at`
- `updated_at`

### 7.2 `assets` 表建议字段

建议至少包含：

- `id`
- `user_id`
- `task_id`
- `file_type`：input/output/preview
- `media_type`：image/video/audio
- `storage_path`
- `file_name`
- `mime_type`
- `file_size`
- `width`
- `height`
- `duration`
- `created_at`

### 7.3 `task_logs` 表建议字段

建议记录：

- 任务状态变化
- 调用 ComfyUI 的请求信息摘要
- 返回结果摘要
- 错误日志

这样排查问题时会轻松很多。

## 8. ComfyUI 对接建议

既然 ComfyUI 是通过 API 使用，那么平台设计时建议重点考虑“如何稳定接入”，而不是“如何在平台里管理工作流”。

### 8.1 平台侧只维护最小映射

建议平台只保存这些信息：

- 某个页面功能对应哪个 ComfyUI 工作流
- 这个工作流需要哪些输入参数
- 这些参数在提交时如何组装

例如：

- 文生图页面 -> `txt2img_workflow`
- 图生视频页面 -> `img2video_workflow`
- 音频生成页面 -> `audio_gen_workflow`

### 8.2 前端不要直接提交原始工作流 JSON

前端最好只提交结构化参数，例如：

```json
{
  "prompt": "a cinematic portrait",
  "negativePrompt": "blur, low quality",
  "width": 1024,
  "height": 1024,
  "seed": 123456,
  "inputImageId": 99
}
```

后端再根据业务类型把这些参数映射到 ComfyUI API 所需结构。

这样做更好维护，也更利于后续替换工作流。

### 8.3 建议优先使用异步任务模式

因为视频和音频任务耗时可能较长，推荐平台采用：

- 提交任务
- 返回平台任务 ID
- 后端异步轮询或监听 ComfyUI 状态
- 前端查看进度与结果

而不是前端一直同步等待结果。

### 8.4 状态同步要统一

建议定义一套平台自己的状态，不直接把 ComfyUI 的原始状态暴露给前端。

这样即使以后更换推理服务，前端也基本不用改。

## 9. 多模态能力接入顺序

虽然目标是支持图片、视频、音频，但还是建议分阶段推进。

### 9.1 第一阶段：图片生成

优先做：

- 文生图
- 图生图
- 局部重绘
- 高清修复

原因：

- 最容易形成完整闭环
- 成果可见性最强
- 对平台基础能力验证最有效

### 9.2 第二阶段：视频生成

再做：

- 图转视频
- 文转视频
- 简单视频增强

因为视频任务：

- 耗时更长
- 文件更大
- 更依赖异步处理与结果管理

### 9.3 第三阶段：音频生成

最后接入：

- TTS
- 配音
- 背景音频生成

这样能避免第一版同时处理三类完全不同的业务细节。

## 10. 部署建议

针对小团队内部平台，建议以简单稳定为主。

### 10.1 第一阶段部署

推荐：

- 1 台应用服务器：前端 + 后端 + MySQL + Redis + MinIO
- 1 台或多台推理服务器：ComfyUI
- 用 `Docker Compose` 统一部署

这套方案对内部平台已经足够。

### 10.2 第二阶段优化

如果后续任务量增加，再考虑：

- 前后端分容器部署
- MySQL 独立部署
- Redis 独立部署
- MinIO 独立部署
- 多个 ComfyUI 实例做负载分担

但一般来说，十几人团队内部使用，未必需要上 `Kubernetes`。

## 11. 推荐开发顺序

建议按下面顺序推进：

### 第一阶段：MVP

- 登录
- 图片生成页面
- 文件上传
- 调用 ComfyUI API 提交任务
- 任务状态查询
- 结果展示与下载
- MySQL 基础表
- MinIO 文件存储

目标是先跑通：

**上传/填写参数 -> 提交任务 -> ComfyUI 执行 -> 返回结果 -> 平台展示**

### 第二阶段：增强版

- 任务列表和筛选
- WebSocket 或轮询进度更新
- 失败重试
- 视频任务接入
- 音频任务接入
- 基础管理后台

### 第三阶段：稳定版

- 日志与告警
- 系统配置后台
- 更完善的权限控制
- 任务统计与使用分析

---

## 12. 最终推荐结论

结合你的实际场景，我建议采用下面这条更轻量、实用的路线：

- **前端**：`Next.js + TypeScript + Tailwind CSS + shadcn/ui`
- **后端**：`NestJS + Prisma + MySQL + Redis`
- **推理对接**：后端直接调用 `ComfyUI API`
- **文件存储**：`MinIO`
- **部署方式**：`Docker Compose`

这条路线的核心特点是：

- 前后端分离清晰
- 平台职责明确，只做任务管理和结果管理
- 不引入不必要的计费、模型中心、工作流后台
- 非常适合 10 人左右团队的内部使用场景

一句话概括：

**把 ComfyUI 当远端推理服务，把平台做成一个轻量的任务管理与结果管理系统，而不是做成一个重型的 AIGC SaaS 平台。**
