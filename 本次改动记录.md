# 本次改动记录

本次主要完成了两部分工作：

- 让后端具备基于 `Prisma + MySQL` 的基础可运行链路
- 让前端任务页具备真实的任务创建表单，并接通后端接口

---

## 1. 后端改动

### 1.1 增加 Prisma 相关脚本

更新文件：`backend/package.json`

新增脚本：

- `prisma:generate`
- `prisma:push`
- `prisma:seed`

作用分别是：

- 生成 Prisma Client
- 将 `schema.prisma` 推送到 MySQL
- 初始化基础数据

### 1.2 优化任务创建逻辑

更新文件：`backend/src/modules/tasks/tasks.service.ts`

主要改动：

- 查询第一个启用状态的用户作为默认任务创建者
- 如果没有种子用户，则抛出错误，提示先执行 seed
- 查询任务列表时一并返回用户基本信息
- 创建任务时把默认状态从 `pending` 调整为 `queued`
- 创建成功后返回关联用户信息

这样做的意义是：

- 不再把 `userId` 写死成 `1`
- 使任务写库逻辑更接近真实可运行状态

### 1.3 增加种子数据脚本

新增文件：`backend/prisma/seed.ts`

初始化内容包括：

- `admin` 角色
- `user` 角色
- 默认管理员用户 `admin`
- 管理员与角色绑定关系
- 一组基础系统配置

当前管理员用户的密码哈希先放了占位值：

- `TEMP_HASH_REPLACE_ME`

这表示当前 seed 主要用于把数据库链路打通，而不是直接投入生产登录。

## 2. 前端改动

### 2.1 新增任务创建面板组件

新增文件：`frontend/components/task-creation-panel.tsx`

这个组件实现了：

- 任务类型选择：图片 / 视频 / 音频

更新文件：`frontend/app/tasks/page.tsx`

主要改动：

- 删除原有静态 mock 列表

- 把数据传给 `TaskCreationPanel`
- 页面职责变成：服务端取数 + 客户端创建任务

这样做之后，任务页已经不再只是展示骨架，而是具备了最基础的“创建任务并入库”的能力。

## 4. 当前项目已经能做到什么

完成这次改动后，项目已经具备以下最小闭环：

1. 前端打开任务页
2. 前端从后端读取任务列表
3. 用户填写任务表单
4. 前端调用后端创建任务接口
5. 后端通过 Prisma 将任务写入 MySQL
6. 前端把新建任务展示到列表中

也就是说，平台现在已经从“纯骨架”进入到了“能跑通基础数据链路”的阶段。

## 5. 如何使用这次新增能力

### 5.1 初始化数据库

先确保 MySQL 中已经有库表。

如果还没有，可以执行：

```bash
mysql -u root -p < init_comfyui_aigc_platform.sql
```

### 5.2 初始化 Prisma

进入 `backend` 目录执行：

```bash
npm run prisma:generate
npm run prisma:push
npm run prisma:seed
```

### 5.3 启动项目

在项目根目录执行：

```bash
npm run dev:backend
npm run dev:frontend
```

启动后访问任务页，即可测试任务创建流程。

## 8. 本次补充修复

在你实际启动后端时，出现了 3 个 TypeScript / Prisma 编译问题，这次已经一并修复。

### 8.1 修复 Prisma `beforeExit` 事件类型报错

更新文件：`backend/src/modules/prisma/prisma.service.ts`

问题原因：

- 当前 Prisma Client 版本下，这里使用 `$on('beforeExit', ...)` 会触发类型不兼容问题

修复方式：

- 删除未实际使用的 `enableShutdownHooks` 方法

影响：

- 不影响当前基础框架运行
- 可以先保证开发环境正常启动

### 8.2 修复 `systemConfigs` 命名不匹配问题

更新文件：`backend/src/modules/system/system.service.ts`

问题原因：

- Prisma 模型名是 `system_configs`
- 代码里错误写成了 `systemConfigs`

修复方式：

- 改为 `this.prisma.system_configs.findMany(...)`

### 8.3 修复 `inputParams` 的 JSON 类型报错

更新文件：`backend/src/modules/tasks/tasks.service.ts`

问题原因：

- Prisma 对 `Json` 字段要求的是 `Prisma.InputJsonValue`
- 原本 DTO 中的 `Record<string, unknown>` 不能直接推断为 Prisma 需要的 JSON 输入类型

修复方式：

- 显式引入 `Prisma`
- 在写库时把 `dto.inputParams` 转为 `Prisma.InputJsonValue`

修复后，后端开发模式应该可以正常继续编译。
