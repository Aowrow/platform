# 资产表排查说明

你提到“资产管理页面能看到上传图片，但数据库里的 `assets` 表没有看到数据”，这个情况我检查过当前项目结构后，结论是：

- **从代码设计上看，`assets` 表结构本身并没有落后于现在的插入逻辑**
- `init_comfyui_aigc_platform.sql` 中的 `assets` 表字段，与当前 Prisma 模型和后端写入逻辑是匹配的

也就是说，这个问题大概率不是“表结构太旧导致插不进去”。

---

## 1. 为什么我判断不是表结构不匹配

当前这三处是对齐的：

### SQL 建表

文件：`init_comfyui_aigc_platform.sql`

`assets` 表包含：

- `user_id`
- `task_id`
- `asset_type`
- `media_type`
- `storage_provider`
- `bucket_name`
- `object_key`
- `file_name`
- `mime_type`
- `file_size`
- `width`
- `height`
- `duration`

### Prisma 模型

文件：`backend/prisma/schema.prisma`

`assets` 模型和上面 SQL 字段是一一对应的。

### 后端写入逻辑

文件：`backend/src/modules/assets/assets.service.ts`

当前插入时写的字段也是：

- `userId`
- `taskId`
- `assetType`
- `mediaType`
- `storageProvider`
- `bucketName`
- `objectKey`
- `fileName`
- `mimeType`
- `fileSize`
- `width`
- `height`
- `duration`

所以从结构上看，没有出现“代码写了新字段，但数据库里没有这个字段”的问题。

---

## 2. 更大的可能是什么

如果“资产管理页能看到图片”，那说明：

- 前端已经请求到了 `GET /api/assets`
- 后端已经通过 Prisma 从 `assets` 表查询到了数据

因为资产管理页的数据来源并不是前端本地缓存，而是后端接口。

所以更大的可能是下面几种情况之一：

1. 你查看的不是当前后端实际连接的那一个数据库
2. 你查看的是另一套环境中的 `assets` 表
3. 你使用的数据库可视化工具还没有刷新
4. 你看的是旧连接，而后端实际连接的是 `backend/.env` 里当前指向的库

当前后端连接的是：

- `backend/.env`
- `DATABASE_URL=mysql://root:202139@localhost:3306/comfyui_aigc_platform`

也就是说，代码层面它正在读写的是：

- `comfyui_aigc_platform.assets`

---

## 3. 这次我做了什么改动

为了让你更直接验证“后端看到的到底是不是数据库里的真实数据”，我新增了一个调试接口：

- `GET /api/assets/debug/summary`

对应文件：

- `backend/src/modules/assets/assets.controller.ts`
- `backend/src/modules/assets/assets.service.ts`

这个接口会返回：

- `assets` 表总数
- 最近 10 条资产记录

这样你不需要猜，也不需要完全依赖前端页面观感，可以直接从后端确认数据库里到底有没有数据。

---

## 4. 推荐你现在怎么验证

请按下面顺序确认：

### 第一步

访问：

- `http://localhost:3001/api/assets/debug/summary`

如果返回了总数和最近资产，说明：

- 后端确实已经从数据库里的 `assets` 表查到了数据

### 第二步

再去你的数据库工具里确认：

- 当前连接的库名是否是 `comfyui_aigc_platform`
- 当前查看的表是否是 `assets`
- 是否刷新了结果集

---

## 5. 当前结论

这次我不建议修改 `assets` 表结构，因为：

- 从当前代码和 SQL 对照来看，结构是匹配的
- 贸然改表不能解决“你看到的数据和后端实际连接的数据不一致”这类问题

所以我选择的方式是：

- **不改表结构**
- **增加调试接口，先确认后端实际查到的数据库数据**

这是一种更稳妥的排查路线。
