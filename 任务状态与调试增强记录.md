# 任务状态与调试增强记录

这次改动主要解决两个实际问题：

1. 前端任务状态和进度不会自动更新
2. 前端提交给 ComfyUI 的最终 prompt 无法直接看到，导致无法确认参数是否真的影响生成

---

## 1. 前端任务状态自动刷新

更新文件：`frontend/components/feature-form.tsx`

这次给功能页加了自动轮询逻辑：

- 当任务状态处于 `queued / running / pending` 时
- 前端每 3 秒自动请求一次 `GET /api/tasks`
- 然后只保留当前功能对应的任务
- 自动刷新任务列表里的状态和进度

这样用户在“分镜图”页面里提交任务后，不需要手动刷新浏览器，也能看到任务从：

- `queued`
- `running`
- `success / failed`

的状态变化。

---

## 2. 前端增加任务调试面板

更新文件：`frontend/components/feature-form.tsx`

新增了一个“任务调试信息”区域，作用是帮助确认：

- 前端输入到底有没有真正传到后端
- 后端最终提交给 ComfyUI 的 prompt 长什么样
- ComfyUI 返回的输出文件信息是什么

当前调试面板支持展示：

- 当前任务状态
- 当前任务进度
- 输出图片数量
- 输出视频数量
- 输出文件列表摘要
- 提交给 ComfyUI 的最终 prompt JSON

这对你当前阶段特别重要，因为你正在验证：

- 前端字段和工作流节点映射是否真的生效

现在你不再需要只凭感觉判断，而是可以直接在前端页面看见实际提交的 prompt。

---

## 3. 后端记录最终提交的 prompt

更新文件：`backend/src/modules/tasks/tasks.service.ts`

这次在任务提交给 ComfyUI 成功后，后端会把以下内容写入 `tasks.resultSummary`：

- `featureCode`
- `queuePosition`
- `submittedPrompt`

同时，也会把这些信息写入任务日志中。

这样做的价值是：

- 你可以追溯“真正发给 ComfyUI 的内容”
- 不是只看前端表单，也不是只看工作流模板原文件
- 而是看最终运行时版本

---

## 4. 后端补充输出文件摘要

更新文件：`backend/src/modules/tasks/tasks.service.ts`

现在轮询完成后，后端会从 ComfyUI 的历史结果中提取更明确的文件信息，写入 `resultSummary`：

- `imageCount`
- `videoCount`
- `files`
  - `filename`
  - `subfolder`
  - `type`

这意味着：

- 即使文件没有按你预期直接出现在某个熟悉的位置
- 你也能先从平台里看到 ComfyUI 回传的输出文件名和子目录

这对排查“为什么 output 文件夹里没看到结果”很有帮助。

---

## 5. 关于“为什么 output 文件夹里看不到结果”

这次还没有直接修改 ComfyUI 的输出机制，但通过调试信息你已经可以先确认两件事：

1. ComfyUI 历史记录里到底有没有返回输出文件
2. 返回的文件路径和文件名具体是什么

如果任务成功但你在 `output` 目录里没看到，常见原因包括：

- 工作流里 `SaveImage` 的 `filename_prefix` 被改到了其他子目录
- 使用的节点不是标准本地图片输出节点
- 某些自定义节点返回的是预览结果，而不是常规 output 文件
- ComfyUI 的输出子目录不是你直觉里看的那个路径

现在因为平台已经把 `files` 摘要展示出来，你可以先根据返回的：

- `subfolder`
- `filename`

去确认 ComfyUI 实际写到了哪里。

这两点一起解决了你当前最关键的验证问题：

## 8. 任务日志策略优化

后来又针对 `task_logs` 做了一次优化，原因是：

- 原先每一次轮询都会写一条 `polling` 日志
- 这会导致一个任务在运行较久时产生很多重复日志

现在已经调整为：

- **只有当轮询状态发生变化时，才写入新的 `polling` 日志**

例如：

- `queued -> running` 会写日志
- `running -> success` 会写最终结果日志
- 如果连续多次轮询结果都还是 `running`，就不会重复插入相同状态的 `polling` 记录

更新文件：`backend/src/modules/tasks/tasks.service.ts`

新增方法：

- `appendTaskLogIfStatusChanged()`

它会先读取当前任务最近一条同类型日志，如果状态没变，就直接跳过写入。

这样做的好处是：

- `task_logs` 仍然保留过程事件设计
- 但不会因为高频轮询产生大量重复数据
- 更适合你这种内部平台的小团队使用场景
