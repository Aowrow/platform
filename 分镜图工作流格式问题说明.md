# 分镜图工作流格式问题说明

这次前端提交后，ComfyUI 返回的核心报错是：

```text
TypeError: argument of type 'int' is not iterable
```

报错位置在：

- `node_replace_manager.apply_replacements(prompt)`

这说明后端提交给 ComfyUI 的 `prompt` 结构不是 ComfyUI API 真正需要的格式。

---

## 1. 问题的根本原因

你放到 `backend/workflows/fenjingtu.json` 里的文件，当前是：

- **ComfyUI 前端编辑器工作流格式**

它的典型特征是：

- 顶层有 `nodes`
- 顶层有 `links`
- 每个节点里有 `pos`
- 每个节点里有 `widgets_values`
- 这是给 ComfyUI 画布编辑器使用的结构

但 `POST /prompt` 需要的不是这个格式，而是：

- **ComfyUI API prompt 格式**

API prompt 格式通常是类似这样的对象：

```json
{
  "3": {
    "class_type": "KSampler",
    "inputs": {
      "seed": 1,
      "steps": 20
    }
  },
  "4": {
    "class_type": "CheckpointLoaderSimple",
    "inputs": {
      "ckpt_name": "xxx.safetensors"
    }
  }
}
```

也就是说：

- `POST /prompt` 需要的是一个“节点 ID -> 节点定义”的映射对象
- 每个节点都必须有：
  - `class_type`
  - `inputs`

而你当前提交的是编辑器工作流 JSON，所以 ComfyUI 在遍历节点时拿到的是数组里的整数或其他编辑器结构，最终触发了：

- `argument of type 'int' is not iterable`

---

## 2. 这次错误和“字段绑定错了”不是一回事

你的怀疑有一部分是对的，但这次最核心的问题还不是“字段绑错节点”，而是：

- **整个工作流 JSON 的格式就不对**

换句话说：

- 现在不是某个 `storyText` 写错地方了
- 而是后端把一份“编辑器格式的 workflow”直接发给了 `POST /prompt`
- 这一步在格式层面就已经不成立了

所以即使字段绑定是对的，只要工作流格式不对，也还是会失败。

---

## 3. 我已经做的修复

更新文件：`backend/src/modules/features/feature-registry.ts`

我加了一层检测：

- 如果读取到的工作流文件顶层存在 `nodes` 和 `links`
- 就直接判定它是“编辑器工作流格式”
- 不再继续往下提交到 ComfyUI
- 而是直接抛出明确错误

现在后端会给出更直接的提示：

- 当前工作流文件不是 API prompt 格式
- 需要重新导出 API 格式后再绑定到平台

这至少能避免你继续看到一大串 ComfyUI 内部 Python 报错，但不知道真正原因。

---

## 4. 正确做法

你需要提供的，不应该是当前这种画布导出 JSON，而应该是：

- **ComfyUI API prompt 格式 JSON**

也就是适合直接提交给：

- `POST /prompt`

的那种结构。

### 你下一步应该怎么做

请把 `fenjingtu.json` 重新导出为 API 可提交格式，然后替换：

- `backend/workflows/fenjingtu.json`

替换后，我就可以继续做两件事：

1. 帮你准确梳理哪些字段可以开放给前端
2. 帮你把这些字段精确绑定到工作流里的对应输入位

---

## 5. 当前结论

这次请求失败的直接原因是：

- **提交给 ComfyUI 的是编辑器工作流，不是 API prompt**

不是这次最主要的问题：

- 前端字段绑定细节是否完全正确

真正的先后顺序应该是：

1. 先把工作流格式换成 API prompt
2. 再确认哪些字段需要绑定
3. 再精细调整前端输入和节点映射

在第 1 步完成前，后面的绑定即使改得再细，也无法稳定跑通。
