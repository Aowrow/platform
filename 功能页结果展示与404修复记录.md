# 功能页结果展示与 404 修复记录

这次改动主要处理了两个问题：

1. “分镜图”功能页出现 404
2. 希望在功能页里直接看到该任务生成出来的图片，而不是必须跳到资产管理页查看

---

## 1. 修复功能页 404 问题

更新文件：

- `frontend/app/features/[code]/page.tsx`
- `frontend/app/page.tsx`
- `frontend/lib/feature-catalog.ts`

这次不仅增加了对路由参数的更稳妥判断，还调整了页面存在性的判定逻辑：

- 在获取 `params` 后先确认 `code` 是否存在
- 如果不存在则直接走 `notFound()`
- 前端新增本地功能目录 `feature-catalog`
- 首页和功能详情页都先使用本地功能目录兜底

这样修改之后，`/features/fenjingtu` 这个页面是否存在，不再依赖后端接口是否成功返回。

也就是说：

- **功能页路由是否存在，应该由前端自己的功能目录决定**
- **后端接口只负责提供运行时数据，而不应该决定这个页面是不是 404**

这也是更合理的设计。

之前之所以会出现 404，本质上是因为：

- 前端把“后端接口拿不到 feature”当成了“页面不存在”

这种设计不适合当前阶段，因为：

- 路由是否存在，应该是前端产品结构问题
- 后端只是在提供可选的配置和运行时数据

现在已经调整为：

- 本地 `feature-catalog` 中存在 `fenjingtu`，页面就能打开
- 后端接口如果正常返回，就优先用后端数据
- 后端接口如果暂时失败，也不会直接把页面打成 404

因此你后续如果再遇到 404，需要优先确认：

- `frontend/lib/feature-catalog.ts` 里是否存在该功能
- 路由路径本身是否正确

## 2. 功能页直接展示生成结果图片

更新文件：`frontend/components/feature-form.tsx`

这次新增了“任务结果图片”区域，支持在功能页中直接查看当前选中任务关联的生成图片。

实现方式：

- 当用户点击某条任务时，前端会请求 `/api/assets`
- 从资产列表中过滤出：
  - `taskId` 等于当前任务 ID
  - `mediaType` 为 `image`
- 然后直接在功能页中展示这些结果图片

这样现在在“分镜图”页中，你可以直接看到：

- 上传的参考图片
- 当前任务的生成结果图片
- 任务状态、进度、调试信息

不需要每次切到资产管理页再确认结果。

## 6. 补充修复：本地功能目录需要包含字段定义

后来又遇到一个前端报错：

```text
Cannot read properties of undefined (reading 'reduce')
```

问题位置在：

- `frontend/components/feature-form.tsx`

根本原因是：

- 前端本地 `feature-catalog` 里只保留了功能基础信息
- 但没有把 `fields` 一并放进去
- 当后端接口临时拿不到 feature 或返回不完整时，页面会回退到本地功能目录
- 这时 `feature.fields` 就会是 `undefined`

### 本次修复方式

更新文件：

- `frontend/lib/feature-catalog.ts`
- `frontend/app/features/[code]/page.tsx`

修复内容：

1. 在本地功能目录中补齐 `fields` 配置
2. 页面在合并本地和后端 feature 数据时，保证：
   - 如果后端返回了 `fields`，优先使用后端
   - 如果后端没有返回 `fields`，回退使用本地 `fields`

这样现在页面即使在后端接口暂时异常或不完整的情况下，也能稳定打开并渲染表单。
