# 云数据库 + 单机部署 + 同机 MinIO 需求说明

## 1. 目标背景

当前项目已经具备：

- 前后端分离基础框架
- 基于功能页的 ComfyUI 工作流调用能力
- 任务状态轮询与结果摘要回写

下一阶段的目标是把项目推进到适合小团队云上使用的形态，采用：

- 云数据库 `MySQL`
- 一台云主机部署前端、后端、ComfyUI
- 同机部署 `MinIO`

这样做的目的：

- 成本可控
- 部署简单
- 便于后续逐步演进
- 从现在开始就建立统一的资源管理方式

---

## 2. 本阶段要达成的能力

### 2.1 统一资源存储能力

需要让平台支持 `MinIO`，用于统一管理：

- 用户上传图片
- ComfyUI 生成图片
- 后续扩展的视频、音频等结果文件

要求：

- 平台不要长期依赖 ComfyUI output 目录作为前端展示来源
- 所有需要在前端展示的资源，最终都要能被平台自身管理

### 2.2 上传图片可管理

需要支持：

- 前端上传图片到后端
- 后端将图片写入 MinIO
- 后端将资源元数据写入 `assets` 表
- 前端可以在资产管理页看到上传的图片

### 2.3 生成图片可管理

需要支持：

- ComfyUI 任务执行完成后，后端解析输出文件
- 后端从本机 ComfyUI output 目录读取文件
- 后端将生成结果同步到 MinIO
- 后端将资源记录写入 `assets` 表
- 前端可以在资产管理页看到生成的图片

### 2.4 前端新增资产管理页

需要新增一个明确的页面，支持：

- 查看上传图片
- 查看任务生成图片
- 显示资源类型、任务来源、创建时间
- 能直接预览图片

---

## 3. 技术路线

## 3.1 存储层

新增 `MinIO` 作为统一对象存储。

建议后端抽象出：

- `StorageService`：负责上传、下载、读取对象

当前阶段只实现 `MinIO`，后续可扩展到：

- `OSS`
- `COS`
- `S3`

## 3.2 资源管理层

使用已有 `assets` 表，统一管理资源元数据。

字段继续沿用当前设计：

- `asset_type`：区分 `input / output / preview / thumbnail`
- `media_type`：区分 `image / video / audio`
- `storage_provider`：当前固定为 `minio`
- `bucket_name`
- `object_key`
- `file_name`
- `mime_type`
- `file_size`
- `task_id`

## 3.3 结果接管层

新增一层“ComfyUI 结果接管”逻辑：

- 轮询完成后解析 `history`
- 识别输出文件名和子目录
- 从本机 ComfyUI output 目录读取文件
- 上传到 MinIO
- 写入 `assets`

## 3.4 前端展示层

前端新增“资产管理”页面，前端不直接访问本地文件路径，而是通过后端提供的统一资源访问地址预览图片。

---

## 4. 需要修改的内容

### 后端

- 增加 MinIO 客户端依赖
- 增加 `storage` 模块
- 增加资源上传接口
- 增强 `assets` 查询返回内容，包含可访问 URL
- 在任务完成时，将 ComfyUI 输出同步到 MinIO 并写入 `assets`
- 增加读取资源内容接口，供前端 `<img>` 使用

### 前端

- 新增资产管理页
- 在导航中增加“资产管理”入口
- 支持查看上传图片和生成图片
- 支持显示资源来源和任务关联信息

### 环境变量

至少需要新增或明确：

- `MINIO_USE_SSL`
- `MINIO_BUCKET`
- `COMFYUI_OUTPUT_DIR`

---

## 5. 本阶段完成后的效果

完成后，平台将具备：

- 图片上传后可在平台内查看
- 分镜图生成后可在平台内查看
- 资源统一进入 MinIO 管理
- 为后续上云给团队使用打下稳定的资源管理基础

---

## 6. 后续演进方向

本阶段完成后，后续可以继续推进：

1. 将图片结果直接展示在功能页里
2. 接入视频与音频资产管理
3. 抽象 `StorageService` 以便切换到云 OSS/COS/S3
4. 为资源增加删除、下载、筛选、标签等能力
