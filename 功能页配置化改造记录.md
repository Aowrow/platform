# 功能页配置化改造记录

这次改动的目标，不再是做一个“通用的图片/视频/音频控制台”，而是改造成：

- 前端以“功能页”为核心
- 每个功能绑定一个独立工作流
- 前端只展示新手真正需要填写的少量字段
- 后端通过配置把字段映射到具体工作流 JSON

这次已经先落地了第一个测试功能：`分镜图`。

---

## 1. 整体思路变化

之前的思路偏向：

- 任务中心
- 通用任务创建
- 按 `image/video/audio` 分类

现在调整为：

- 能力首页
- 按功能进入页面
- 每个功能绑定一个工作流文件
- 功能表单由配置自动渲染

这更适合：

- 内部使用
- 面向新手
- 明确业务目标的 AIGC 工具

---

## 2. 后端改动

### 2.1 新增功能注册表

新增文件：`backend/src/modules/features/feature-registry.ts`

这是本次改造的核心。

当前它负责：

- 定义有哪些功能
- 为每个功能定义前端表单字段
- 指定对应工作流文件
- 负责把前端输入映射到工作流 JSON

当前已经注册了：

- `fenjingtu`

并且绑定了：

- `backend/workflows/fenjingtu.json`

### 2.2 新增功能模块

新增文件：

- `backend/src/modules/features/features.module.ts`
- `backend/src/modules/features/features.service.ts`
- `backend/src/modules/features/features.controller.ts`
- `backend/src/modules/features/dto/create-feature-task.dto.ts`

新增接口：

- `GET /api/features`
- `GET /api/features/:code`
- `POST /api/features/:code/tasks`

作用：

- 前端可以获取当前有哪些功能
- 前端可以根据功能配置自动渲染表单
- 提交任务时不再直接按通用媒体类型处理，而是按功能处理

### 2.3 重构任务创建逻辑

更新文件：`backend/src/modules/tasks/tasks.service.ts`

现在创建任务的核心入口已经变成：

- `createForFeature(featureCode, dto)`

执行流程是：

1. 根据 `featureCode` 找到功能定义
2. 加载对应工作流 JSON
3. 通过功能映射规则写入表单参数
4. 写入平台任务表
5. 提交到 ComfyUI
6. 轮询状态
7. 回写数据库

### 2.4 简化 ComfyUI 服务层

更新文件：`backend/src/modules/comfyui/comfyui.service.ts`

改动重点：

- 移除了之前内置的 `txt2img` 工作流构造逻辑
- 现在 `ComfyuiService` 只负责：
  - 提交 prompt
  - 查询 queue
  - 查询 history

也就是说：

**工作流构造不再放在 ComfyUI 服务里，而放在功能注册表里。**

这更符合“一个功能绑定一个工作流”的方向。

### 2.5 增加分镜图工作流映射

当前 `fenjingtu` 的映射逻辑已经先做了一版可测试实现：

- 修改剧情文本输入
- 修改视觉风格补充语
- 修改输出比例
- 修改保存前缀
- 可选覆盖工作流中的模型 API Key 和 Base URL

这里是基于你现有 `fenjingtu.json` 的节点结构做的定向映射。

后续如果你换了新工作流，只需要更新对应 feature 的映射逻辑即可。

## 3. 前端改动

### 3.1 首页从“总览”改成“能力首页”

更新文件：`frontend/app/page.tsx`

现在首页会调用：

- `GET /api/features`

并把所有功能渲染成能力卡片。

### 3.2 导航改成能力导向

更新文件：`frontend/components/app-shell.tsx`

这次删除了原来偏通用控制台的入口：

- `任务中心`
- `资源中心`

改成：

- `能力首页`
- `分镜图`
- `系统配置`

### 3.3 新增功能卡片组件

新增文件：`frontend/components/feature-card.tsx`

用于在首页展示功能入口。

### 3.4 新增通用功能表单组件

新增文件：`frontend/components/feature-form.tsx`

这个组件是“第二种方案”的关键部分。

它支持：

- 根据后端返回的字段配置自动渲染表单
- 提交到 `POST /api/features/:code/tasks`
- 展示该功能最近任务

这意味着后续新增功能时，前端不一定要重复写一个新表单组件。

### 3.5 新增功能详情页路由

新增文件：`frontend/app/features/[code]/page.tsx`

它会：

- 根据功能编码加载功能配置
- 读取该功能相关任务
- 使用通用 `FeatureForm` 渲染页面

当前你已经可以访问：

- `/features/fenjingtu`

作为“分镜图”测试页。

## 7. 当前已经达成的效果

1. 首页展示功能列表
2. 进入“分镜图”功能页

这条链路比之前更接近你真正想要的产品方向。

## 9. API Prompt 格式替换后的修正

你把 `backend/workflows/fenjingtu.json` 替换成 ComfyUI API prompt 格式后，我同步检查并修正了：

- `backend/src/modules/features/feature-registry.ts`

### 9.1 为什么之前那版必须改

之前那版映射逻辑是按“编辑器工作流格式”写的，使用的是：

- `workflow.nodes`
- `widgets_values`

而你现在换成的是 API prompt 格式，结构已经变成：

- `workflow[节点ID].inputs.xxx`

所以旧版映射方式已经不再适用，必须改成按 `inputs` 写值。

### 9.2 现在分镜图功能实际改写的节点

当前已经按新的 API prompt 结构改成：

- 节点 `40` 的 `inputs.string_b`
  - 写入前端的“剧情文本”
- 节点 `46` 的 `inputs.aspect_ratio`
  - 写入前端的“输出比例”
- 节点 `45` 的 `inputs.filename_prefix`
  - 改成平台自己的输出路径前缀
- 节点 `47` 的 `inputs.string_a`
  - 追加“视觉风格 / 分镜布局 / 输出比例”补充语
- 节点 `9` 和 `55` 的 `inputs.api_key/base_url`
  - 如果环境变量中存在，就覆盖工作流里的值

### 9.3 当前判断

从你现在这份 API prompt 看，当前映射逻辑已经比之前合理很多，因为它确实是在修改 ComfyUI API 真正会读取的 `inputs` 字段。

不过需要说明：

- 现在这版仍然是“基于工作流结构推断出的第一版正式映射”
- 是否完全符合你的业务意图，还取决于你最终希望前端控制哪些输入位

换句话说：

- 这次已经不是“格式错误”了
- 现在进入的是“业务字段和节点映射是否精确”的阶段
